## Merchant Dashboard

Merchant-facing analytics dashboard with row-level scoping per authenticated merchant.

### Required env

Create `.env` with:

```bash
DATABASE_URL=postgres://user:pass@host:5432/dbname
AUTH_SESSION_SECRET=replace-with-random-long-secret
```

### Database setup

1. Run migration SQL:

`lib/db/migrations/001_auth_tables.sql`

2. Seed a merchant user mapping using:

`scripts/seed-merchant-user.sql`

For `password_hash`, use format generated by `scrypt`: `salt_hex:hash_hex`.

3. Create canonical merchant mapping table (groups multiple `merchant_key` per `uniq_merchant`):

`scripts/create-merchant-canonical-map.sql`

4. If one merchant brand has multiple `merchant_key` values, normalize to one canonical key using:

`scripts/normalize-canonical-merchant-key.sql`

Canonical grouping key is `dim_merchant.uniq_merchant`.

5. (Optional) Set `users.username` from `dim_merchant.uniq_merchant` for merchant logins:

`scripts/update-username-from-uniq-merchant.sql`

6. (Optional) Auto-create missing merchant users from canonical mapping:

`DATABASE_URL=... node scripts/create-merchant-users.ts`

### Auth flow

- `POST /api/auth/login` sets secure httpOnly cookie session.
- `POST /api/auth/logout` clears session cookie.
- `GET /api/auth/session` returns merchant identity from session.

### Security model

- Dashboard APIs resolve merchant context from session -> `users` + `merchant_users`.
- API queries enforce `merchant_key = mapped merchant_key`.
- Client-provided merchant identity is ignored.

### Merchant dashboard routes

- `/login`
